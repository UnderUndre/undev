# Review: DevOps Dashboard Spec (v1.0)
## 🛠 Обзор от Валеры (Digital Plumber)

Спека прочитана. Идея понятна — натянуть гуй (GUI) на проверенные bash-скрипты через SSH. Это снизит порог входа и добавит визуализации. Но есть узкие места, где трубы могут протечь. 

### 🛑 Критические дыры и вопросы (Что рванет первым)

1. **Мониторинг через SSH (FR-030, FR-031)**
   - **Проблема:** Дергать `health-check.sh` по SSH каждые 60 секунд на каждый сервер — это изнасилование `auth.log` и оверхед на рукопожатия TCP/SSH.
   - **Вопрос:** Будет ли использоваться SSH multiplexing (ControlMaster) для удержания соединения, или мы будем плодить короткоживущие сессии? 
2. **Парсинг вывода баша (FR-050, FR-030)**
   - **Проблема:** "parse structured output". Баш скрипты сыпят текстом, цветами (ANSI), варнингами. 
   - **Вопрос:** Умеют ли скрипты `@underundre/undev` отдавать строгий JSON (`--json` флаг)? Если нет парсить stdout регулярками — это гарантированная головная боль.
3. **Deployment Lock (FR-017)**
   - **Проблема:** "Только один деплой на сервер (deployment lock)". Если дашборд упадет (OOM, рестарт) или отвалится сеть в момент деплоя — лок зависнет навсегда.
   - **Решение:** Нужен механизм TTL (time-to-live) для лока и/или кнопка "Force Unlock" (исключительно с записью в аудит-лог).
4. **Хранение секретов (FR-003)**
   - **Проблема:** "encrypted at rest". Дашборд работает в докере.
   - **Вопрос:** Откуда берется мастер-ключ для расшифровки приватных SSH-ключей в БД? Из ENV-переменной при старте контейнера? Спека об этом молчит.
5. **Определение успеха деплоя (US-001)**
   - **Проблема:** Как дашборд понимает, что деплой успешен? По exit code 0 от `deploy.sh`? А если докер поднялся, но приложение внутри крашится в CrashLoopBackOff?
   - **Вопрос:** Если скрипт `deploy.sh` сам ждет health-check приложения перед отдачей success — отлично, но это должно быть явным контрактом.
6. **Отсутствие таймаутов на операции**
   - **Проблема:** В NFR есть таймауты SSH (30s), но нет Time-To-Live для самого деплоя. Если `npm install` зависнет бесконечно ожидая ввод, дашборд положит процесс или зависнет на 24 часа? Нужен хард-таймаут на выполнение (например 15-20 минут).

### 📝 Что не указано (Assumptions & Missing Features)

- **Механика отката (US-002):** Ожидается откат "за пару минут". Это предполагает симлинки (Capistrano-style) на сервере или запуск сохраненных docker images. Поддерживают ли это скрипты `@underundre/undev`?
- **Очистка стейта дашборда:** Написано про ретеншн БД (14 дней, FR-024), но нет ретеншна для аудит-логов и истории деплоев внутри самого дашборда. Со временем его собственная база раздуется.
- **WebSocket Reconnection:** При скачке сети (NFR-005 - mobile browsers) вебсокет отвалится. Механизма переподключения и докачки потерянных логов (US-005) в требованиях нет.

### 🎯 Резюме (Understanding Lock)

**Суть:** Удобный, self-hosted дашборд-wrapper над готовыми bash-флоу для соло-девов/небольших команд.
**Ограничения:** Зависит от стабильности работы скриптов по сети.
